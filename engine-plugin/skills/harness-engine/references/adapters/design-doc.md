# Design Document Task Adapter

## 목적

harness-engine이 설계 문서 도메인 하네스를 생성하거나 보강할 때, 공통 `research` phase 이후 이 어댑터를 추가 적용하여 설계 문서 task_type 고유의 **최소 계약**을 적용한다.

설계 문서에는 RFC, ADR(Architecture Decision Record), 시스템 설계서, API 설계서, 마이그레이션 계획 등이 포함된다.

## Coverage Contract

설계 문서 하네스에는 다음 항목이 반드시 포함되어야 한다. 누락 시 하네스가 불완전한 것으로 판정한다.

### 필수 축 (모든 설계 문서)

1. **문제 정의**
   - 현재 상태(As-Is)와 문제점
   - 변경의 동기(trigger) — 무엇이 이 설계를 필요하게 했는가
   - 성공 기준 (측정 가능한 형태)
   - 비기능 요구사항 (성능, 보안, 확장성, 비용 등)
2. **대안 분석**
   - 최소 3개 대안 (현상 유지 포함)
   - 대안별 장단점 비교 매트릭스
   - 각 대안의 리스크와 비용 추정
   - 기각 사유 명시
3. **결정 기록**
   - 선택된 대안과 선택 근거
   - 결정 참여자/이해관계자
   - 결정 일자
   - 재검토 조건 (어떤 상황이 되면 이 결정을 다시 평가하는가)
4. **구현 계획**
   - 단계별 마일스톤
   - 의존성과 선행 조건
   - 점진적 롤아웃 전략 (해당 시)
   - 예상 일정 범위
5. **롤백/복구 계획**
   - 실패 시나리오 정의
   - 롤백 절차
   - 데이터 복구 전략 (해당 시)
   - 모니터링/알림 기준

### 프로젝트 contract packet에서 반드시 조사할 항목

- 문서 유형 (RFC, ADR, 시스템 설계, API 설계, 마이그레이션)
- 대상 독자 (엔지니어, 경영진, 교차팀)
- 기존 아키텍처/시스템 현황
- 조직의 의사결정 프로세스 (승인 필요 여부)
- 기존 설계 문서 템플릿/규약 유무

## 1차 근거 소스

설계 문서 하네스의 규칙은 다음 소스를 1차 근거로 사용한다:

1. 대상 기술의 공식 문서 (아키텍처 가이드, 모범 사례)
2. 표준 설계 패턴 (GoF, DDD, CQRS, Event Sourcing 등 해당 시)
3. 조직의 기존 ADR/RFC 아카이브
4. 학술 자료 (분산 시스템, 보안 등 해당 영역)

보조 자료:
- Context7 MCP: 기술 스택별 문서 빠른 참조
- Sequential Thinking MCP: 복잡한 대안 분석 구조화

## Anti/Good 최소 필수 쌍 목록

### 문제 정의

| 케이스 | Anti 내용 | Good 내용 |
|---|---|---|
| 솔루션 선행 | "X를 도입한다"로 시작, 왜 필요한지 미기재 | 현재 문제 → 동기 → 성공 기준 순서로 서술 |
| 모호한 성공 기준 | "성능을 개선한다" | "P95 응답 시간 200ms 이하, 처리량 1000 RPS 이상" |
| 범위 누락 | 기능 요구만 기술, 비기능 요구 없음 | 성능, 보안, 비용, 운영 복잡도 등 비기능 요구 명시 |

### 대안 분석

| 케이스 | Anti 내용 | Good 내용 |
|---|---|---|
| 단일 대안 | 선택할 기술 하나만 설명 | 최소 3개 대안 (현상유지 포함) + 비교 매트릭스 |
| 편향된 비교 | 선호 대안의 장점만 부각 | 각 대안의 장단점, 리스크, 비용을 동등하게 분석 |
| 기각 사유 없음 | 대안을 나열만 하고 왜 선택 안 했는지 미기재 | 기각된 대안별 구체적 기각 사유 명시 |

### 결정 기록

| 케이스 | Anti 내용 | Good 내용 |
|---|---|---|
| 암묵적 결정 | 결정 근거 없이 구현 방향만 서술 | 명시적 결정문 + 근거 + 참여자 + 일자 |
| 영구 결정 | 재검토 조건 없는 절대적 결정 | 재검토 트리거 조건 명시 ("트래픽 10배 시 재검토") |

### 구현 계획

| 케이스 | Anti 내용 | Good 내용 |
|---|---|---|
| 빅뱅 배포 | 전체 시스템을 한 번에 전환 | 단계별 마일스톤, 점진적 롤아웃 (카나리, 블루-그린 등) |
| 의존성 무시 | 선행 조건 미확인 상태로 일정 수립 | 의존성 그래프, 크리티컬 패스 식별 |

### 롤백/복구

| 케이스 | Anti 내용 | Good 내용 |
|---|---|---|
| 롤백 계획 부재 | "문제 없을 것" 가정으로 롤백 미계획 | 실패 시나리오별 롤백 절차 + 데이터 복구 전략 |
| 모니터링 없는 배포 | 배포 후 수동 확인에 의존 | 자동 알림 기준 + 롤백 트리거 정의 |

## 드라이런 입출력 예시

### Positive Case: RFC 구조 검증

**Input**: "마이크로서비스 인증 시스템을 모놀리스에서 분리하는 설계 문서를 작성해줘."

**Expected Output (하네스가 안내해야 하는 방향)**:

- 현재 모놀리스 인증의 문제점 정의 먼저 요구
- 최소 3개 대안 (모놀리스 유지, 인증 서비스 분리, BFF 패턴 등)
- 대안별 비교 매트릭스 (복잡도, 보안, 지연시간, 운영 비용)
- 마이그레이션 단계 계획 + 롤백 전략

### Negative Case: 솔루션 선행 감지

**Input**: "Kafka를 도입하는 설계 문서를 작성해줘."

**Expected Output (하네스가 차단해야 하는 패턴)**:

- 솔루션 선행 → Anti-pattern으로 감지
- "왜 Kafka가 필요한지" 문제 정의부터 요구
- 대안 (RabbitMQ, SQS, 현상유지 등) 비교 요구

## 문서 유형별 분기

설계 문서 유형에 따라 필수 축의 강조점이 달라진다:

| 유형 | 강조 축 | 완화 가능 축 |
|---|---|---|
| **RFC** | 대안 분석, 결정 기록 | 세부 구현 계획 (RFC 승인 후 별도 작성) |
| **ADR** | 결정 기록, 재검토 조건 | 상세 구현 계획 (ADR은 결정 기록이 핵심) |
| **시스템 설계** | 구현 계획, 롤백/복구 | — (모든 축 필수) |
| **API 설계** | 인터페이스 계약, 하위호환성 | 내부 구현 세부사항 |
| **마이그레이션 계획** | 구현 계획, 롤백/복구, 데이터 전환 | — (모든 축 필수) |

contract packet에서 문서 유형을 먼저 확정하고, 해당 유형의 강조 축에 집중한다.

## 설계 근거

- Coverage Contract 필수 축: Google Design Docs, Stripe RFC 프로세스, ADR(Architecture Decision Record) 표준 구조 교차 분석
- 대안 분석 최소 3개: 인지 편향 방지를 위한 표준 의사결정 프레임워크 (현상유지 포함은 "do nothing" 대안 강제)
- Anti/Good 필수 쌍: 실제 설계 리뷰에서 반복적으로 발생하는 결함 패턴 기반
